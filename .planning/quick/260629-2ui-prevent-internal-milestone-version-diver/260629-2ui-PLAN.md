# Quick Task 260629-2ui

## Problem

GSD milestones are free-form `vX.Y` text, so the project ran a parallel
"internal" milestone line (v1.0..v1.3) unrelated to the real shipped plugin
version (4.x) — forcing everyone to track two numbers ("v1.3 shipped as plugin
v4.0.0"). Nothing detected or prevented the divergence. The single-version-line
decision (2026-06-29) needs a durable guardrail so it can't silently recur.

## Solution

Add a version-alignment detector following the existing drift-detector pattern
(matches the project's established CI guard philosophy; `.planning/` docs are
force-tracked, so CI can read the milestone version).

### Tasks

1. `bin/maintenance/check-version-alignment.cjs` — compares the product version
   (`.claude-plugin/plugin.json`) against the active milestone version
   (`.planning/STATE.md` frontmatter, fallback MILESTONES.md). Passes when the
   milestone major equals the plugin major (current line) or plugin major + 1
   (next major in progress); fails on a parallel line. Also verifies
   plugin.json/marketplace.json agree. SKIPs when no manifest or no milestone
   (general/multi-project safe). Escape hatch: `VERSION_ALIGNMENT_ALLOW=1` or a
   committed `.version-alignment-allow` file. Exports pure helpers for tests.
2. `tests/version-alignment.test.cjs` — zero-dep unit tests for the comparison.
3. Wire into `bin/maintenance/check-drift.cjs` umbrella + a `version-alignment`
   CI job in `.github/workflows/check-drift.yml`.

## Verification

- `node tests/version-alignment.test.cjs` → all pass
- `node bin/maintenance/check-version-alignment.cjs` → PASS (v4.1 ↔ 4.0.1)
- Divergent v1.3-vs-4.x scenario → exits 1 with a clear remediation message
