<transform_note>
README.md flows through the bm byte-transform (`bin/build-bm.cjs`), which rewrites
every `/gsd:` (not followed by a slash) to `/bm:` in the generated `dist/bm/README.md`.
Two authoring rules follow, and they differ by content type:
- Runnable command EXAMPLES: write them DIRECTLY as `/bm:X` in this source README
  (e.g. `/bm:new-project`). Do NOT write them as `/gsd:X` and rely on the byte-transform
  to flip them. D-04 and ROADMAP success criterion #1 require the AUTHORED README to
  document /bm: as the primary command surface, so the source itself must read `/bm:`.
- Migration and identity SENTENCES (the retirement-date prose and the "gsd keeps
  working" prose) must read correctly in BOTH packages. Write those using the bare word
  `gsd` with no colon and no slash (the transform never matches bare `gsd`). Example
  that reads correctly in both packages: "The original gsd command prefix retires at
  v5.0 on 2026-10-01." Do NOT write "/gsd: retires ..." inside a sentence, or the bm
  README will say "/bm: retires ..." (factually wrong).
Net rule:
- Migration/identity sentences -> bare word `gsd` (no colon), never `/gsd:` or `/bm:`.
- Runnable command examples -> write `/bm:X` directly in source.
- Never depend on the byte-transform to flip authored README command examples.
The generated bm prose is read by eye in plan 15-05; keeping migration sentences
bare-word is what makes that read pass.
</transform_note>
