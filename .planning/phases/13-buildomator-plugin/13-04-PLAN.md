Detector design constraints (why a raw-text, per-class detector - not pre-stripping):
- Two violation classes are substring-supersets of allow-listed tokens, so pre-removing the
  allow-list BEFORE matching destroys the very violation it should catch:
    * 'cache/gsd-plugin/gsd' contains the allow-listed 'gsd-plugin'; stripping 'gsd-plugin'
      leaves 'cache//gsd' and the CR-01 violation is gone.
    * 'gsd:gsd-<x>' contains an allow-listed 'gsd-<alnum>' filename shape; stripping leaves
      'gsd:' and the D-08 agent-ref shape is gone.
  Therefore each class is matched by a DIRECT, anchored pattern against the RAW (unstripped)
  text; allow-list sparing is expressed WITHIN each pattern (the char after the colon),
  mirroring how rewriteCommandRefs's gsd:(?!/) already spares gsd://.
- Verified against source: mcp/server.cjs (NOT in STAMP_EXCLUDE, so the census scans it) has
  ZERO gsd:[a-z] occurrences; its only gsd: tokens are URI-form gsd:// (colon-slash) and the
  regex-escaped gsd:\/ (colon-backslash) at lines 470/477 - both spared by the [a-z]-after-colon
  requirement. This is what lets the namespace-prefix class use gsd:[a-z] safely across the
  whole non-excluded tree.

D-08 census allow-list (tokens that legitimately contain gsd and must NOT be flagged):
- gsd-plugin (marketplace id)
- gsd-<alnum> filenames (gsd-tools.cjs, gsd-sdk, gsd-session-state.sh, gsd-validate-commit.sh, etc.)
- gsd:// and the regex-escaped gsd:\/ URI form
- gsd_<alnum> (MCP tool names)
- camelCase gsd[A-Z] identifiers
- open-gsd / opengsd
- .gsd (dotfile/dir) and the gsd/{milestone} branch template
- MCP internal name:'gsd', the plugin.json gsd keyword, and branding GSD / Get Shit Done / [gsd]

Violation classes the detector MUST flag (post-transform, these should never appear in
non-excluded dist/bm files):
- cache/gsd-plugin/gsd (un-stamped hook fallback)
- gsd:gsd-<x> (un-rewritten agent ref)
- /gsd:<x> and name: gsd:<x> (un-rewritten slash command or frontmatter)
- /gsd[:-]<x> (un-rewritten sanitizer literal)
