## Summary

This phase introduces the repo's FIRST build/generate step. Today the repo root IS the installed `gsd` plugin (`marketplace.json` source `"./"`). Phase 12 adds a generator that copies the authored `gsd` source into `dist/bm/`, stamps only the identity/branding fields (`name` gsd->bm, `description`/branding -> Buildomator, and `displayName` -> "Buildomator"), and adds a second `bm` entry to the single `marketplace.json` whose `source` is `"./dist/bm"`. Version is single-sourced from `.claude-plugin/plugin.json` and copied to every manifest so all four version sites (gsd plugin.json, gsd marketplace entry, bm plugin.json, bm marketplace entry) match. A CI test regenerates `dist/bm/` and diffs it against the committed copy, failing on any drift outside the whitelisted identity fields. The install-smoke workflow is extended to exercise the bm package path.

The genuinely uncertain mechanics are now RESOLVED against current Claude Code docs (code.claude.com, fetched 2026-07-02):
- One `marketplace.json` can list multiple plugins, each with its own relative `source` that resolves against the marketplace root at the installed git ref — so the committed `dist/bm/` WILL resolve for bm installs (validates D-04). [CITED]
- Cache layout is `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/`; each plugin gets its own copy and its own `${CLAUDE_PLUGIN_ROOT}` (validates D-10). [CITED]
- The command prefix and all component namespacing derive from the plugin `name` — stamping `name` is the entire divergence lever for the command surface (validates D-02/D-09). [CITED]

**Primary recommendation:** Write one Node build script (`bin/build-bm.cjs`) that recursively copies the whitelisted source tree into `dist/bm/`, JSON-stamps the two manifests, single-sources version from `plugin.json`, and exposes a `--check` mode that CI runs to diff a fresh build against the committed `dist/bm/`. Add the `bm` marketplace entry with `"source": "./dist/bm"`. Do NOT use symlinks for the package payload. The single highest-risk landmine is the hardcoded `.claude/plugins/cache/gsd-plugin/gsd` fallback path baked into every hook command in `hooks/hooks.json` — it will make the bm plugin's hooks fall back to the gsd cache dir. Decide its handling explicitly (see Pitfall 1).

### Pattern 1: Generate-and-Stamp (copy the source, mutate only identity fields)
**What:** A build script recursively copies the authored source into `dist/bm/`, then applies a small whitelist of field mutations to the copied `plugin.json`. Everything else is byte-identical.
**When to use:** This phase's entire mechanism (D-02).
**Example:**
```javascript
// bin/build-bm.cjs  (pattern sketch — planner writes the real thing)
'use strict';
const fs = require('fs');
const path = require('path');

const ROOT = process.cwd();
const OUT = path.join(ROOT, 'dist', 'bm');

// Directories/files that must NEVER enter the package payload.
const EXCLUDE = new Set(['.git', '.planning', '.claude', 'node_modules',
  'dist', 'scratchpad', '.DS_Store']);

function filter(src) {
  const base = path.basename(src);
  return !EXCLUDE.has(base);
}

// 1. Single-source identity + version from the authored manifest.
const srcManifest = JSON.parse(
  fs.readFileSync(path.join(ROOT, '.claude-plugin', 'plugin.json'), 'utf8'));
const version = srcManifest.version;

// 2. Clean + recursive copy (Node >=16.7).
fs.rmSync(OUT, { recursive: true, force: true });
fs.cpSync(ROOT, OUT, { recursive: true, filter });

// 3. Stamp ONLY the whitelisted identity/branding fields.
const bmManifest = { ...srcManifest };
bmManifest.name = 'bm';                       // command prefix → /bm:
bmManifest.displayName = 'Buildomator';       // UI label (CC >= 2.1.143)
bmManifest.description =
  'Buildomator -- structured workflow plugin for Claude Code ...';
bmManifest.version = version;                 // lockstep (redundant but explicit)
fs.writeFileSync(
  path.join(OUT, '.claude-plugin', 'plugin.json'),
  JSON.stringify(bmManifest, null, 2) + '\n');

// NOTE: the generated dist/bm should NOT itself contain a marketplace.json
// that lists plugins (the single ROOT marketplace.json owns both entries).
// If dist/bm inherits a copy of marketplace.json, either delete it in the
// build or leave it inert — CC only reads the marketplace.json at the
// marketplace root, not inside an individual plugin source.
```
Source pattern basis: [VERIFIED: codebase — mirrors `bin/validate-plugin.cjs` + `bin/maintenance/check-version-alignment.cjs` structure]

**When to use:** CI job in `check-drift.yml` (D-06). Also a `tests/build-bm-drift.test.cjs` unit test for the pure stamp/whitelist logic (following the `evaluateAlignment` export-pure-helper pattern).

```javascript
// Fail listing any path that differs. Model the harness on
// tests/version-alignment.test.cjs (node:assert, check(), exit 1 footer).
```

### Pattern 3: Single Marketplace, Two Plugin Entries
**What:** One `.claude-plugin/marketplace.json` (name `gsd-plugin`) lists both `gsd` (`source: "./"`) and `bm` (`source: "./dist/bm"`). Each relative source resolves against the marketplace ROOT at the installed git ref. [CITED: code.claude.com/docs/en/plugin-marketplaces — "Paths resolve relative to the marketplace root ... work when users add your marketplace from a git source"]
**Example (target marketplace.json shape):**
```json

  "owner": { "name": "Jasper Nuyens" },

    { "name": "bm",  "source": "./dist/bm", "version": "4.1.0",

```

### Anti-Patterns to Avoid
- **Symlinking the bm payload to the repo root:** CC dereferences same-marketplace symlinks by copying content, but a whole-root symlink recurses into `dist/` and can't carry a stamped manifest. Use copy + stamp.
- **A second full `marketplace.json` inside `dist/bm/`:** only the marketplace-root file is read for the plugin catalog. A copied-in `marketplace.json` under `dist/bm/` is inert at best; ensure the build doesn't create a self-referential nested marketplace. Delete or ignore it in the build.
- **Stamping anything beyond name/displayName/description/version:** widens the divergence surface and breaks success criterion 4. Keep the whitelist tiny; the drift guard enforces it.
- **Copying `.planning/`, `.claude/`, `.git/`, `node_modules/`, `dist/`, `scratchpad/` into `dist/bm/`:** bloats the package and can create a recursive `dist/bm/dist/bm/...`. The copy filter MUST exclude these.
- **Introducing a plugin or marketplace name that collides:** never name a plugin `gsd-plugin` nor a marketplace `gsd`/`bm` (D-11, CC #34200 ENAMETOOLONG).

## Common Pitfalls

### Pitfall 1: Hardcoded gsd cache fallback in hooks (BUILD-03 landmine)
**What goes wrong:** Every hook command in `hooks/hooks.json` contains an inline Node resolver whose fallback is literally `path.join(o.homedir(),'.claude/plugins/cache/gsd-plugin/gsd')`. When this file is copied unchanged into `dist/bm/`, the bm plugin's hooks — if `${CLAUDE_PLUGIN_ROOT}` is set (normal case) — resolve correctly to the bm root. But if `${CLAUDE_PLUGIN_ROOT}` is ever unset or the baked path is pruned, the bm hooks fall back to the **gsd** cache dir (`cache/gsd-plugin/gsd/...`), running gsd's code, not bm's.
**Why it happens:** The fallback was written (commit 7a80d47, 260420-vfb) to survive a pruned `${CLAUDE_PLUGIN_ROOT}` for the single gsd plugin; it assumes exactly one plugin named `gsd`.
**How to avoid (Phase 12 scope):** Because D-02 forbids editing anything but identity/branding, and because both packages are byte-identical here, the SAFE Phase-12 position is: the primary path (`${CLAUDE_PLUGIN_ROOT}` set) is correct for bm, and the fallback running gsd's identical code is harmless while the two are byte-identical. VERIFY this with the extended install-smoke test (assert bm hooks resolve via `${CLAUDE_PLUGIN_ROOT}`, not the fallback). Flag the fallback for a proper per-plugin fix in Phase 13/14 once the packages diverge (once bm has `/bm:` docs, running gsd's fallback code would be wrong). Document this as a known limitation.
**Warning signs:** bm session shows `GSD: plugin path stale, using .../gsd/...` on the stderr line — that message means the fallback fired.

### Pitfall 2: Recursive copy self-inclusion (`dist/bm/dist/bm/...`)
**What goes wrong:** If the copy filter doesn't exclude `dist/`, `fs.cpSync(ROOT, dist/bm)` recurses into the growing output.
**How to avoid:** Exclude `dist` (and `.git`, `.planning`, `.claude`, `node_modules`, `scratchpad`) in the `filter` predicate. Clean `dist/bm` with `fs.rmSync(..., {recursive, force})` before each build for determinism.
**Warning signs:** Build hangs or `dist/bm` balloons; drift test never converges.

### Pitfall 3: Marketplace source doesn't resolve because dist/bm isn't committed
**What goes wrong:** If `dist/bm/` is gitignored or not committed, a user installing `bm@gsd-plugin` from the GitHub marketplace gets a source-not-found error — CC resolves the relative `source` against the marketplace copy at the installed ref, which is the committed tree. [CITED]
**How to avoid:** Commit `dist/bm/` as part of every release (D-04/D-05). Ensure `.gitignore` does NOT list `dist/`. (Current `.gitignore` ignores `.planning/`, `.claude/`, `node_modules/`, `.DS_Store` — `dist/` is NOT ignored, so committing works today. Do not add it.) [VERIFIED: codebase — `.gitignore` read]
**Warning signs:** `plugin-not-found` / source path error on bm install; install-smoke passes locally (uncommitted tree) but fails from a clean checkout.

### Pitfall 4: Version drift across the four manifest sites
**What goes wrong:** A release bumps `plugin.json` but forgets one of: gsd marketplace entry, bm plugin.json, bm marketplace entry.
**How to avoid:** Single-source version from `.claude-plugin/plugin.json` in the build (D-08); the build writes all copies. Extend `check-version-alignment.cjs` to assert the bm marketplace entry version == plugin version (it currently checks only the first entry with a `version`). The drift guard re-running the build catches a stale committed `dist/bm/`.
**Warning signs:** CC treats refs with equal version strings as identical and skips updates; a mismatched bm version could ship stale.

### Pitfall 5: Both plugins enabled -> hooks/MCP fire twice
**What goes wrong:** Each plugin gets its own `${CLAUDE_PLUGIN_ROOT}` and its own hook registration; enabling both `gsd` and `bm` fires every PostToolUse/SessionStart hook twice and starts two MCP `stdio` servers (both named for their plugin). [CITED: each plugin loads independently] This is CONTEXT D-10 and REQUIREMENT COMPAT-02/03 — explicitly DEFERRED to Phase 14.
**How to avoid (Phase 12):** Do NOT solve dedup here. But be aware install-smoke should test the bm path in isolation (bm enabled, or bm path directly), not assert single-fire behavior. Note the deferral so the plan doesn't accidentally scope it in.
**Warning signs:** Double checkpoint writes, duplicate MCP registration — expected when both enabled; out of scope until Phase 14.
